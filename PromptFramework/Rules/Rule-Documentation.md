# Rule — Documentation

## Objetivo

Establecer las reglas para la generación de documentación técnica que represente fielmente el conocimiento del proyecto, facilite su comprensión y permita operar, mantener, evolucionar o reconstruir el sistema documentado.

---

# Principios generales

La documentación constituye la fuente principal de conocimiento del proyecto.

Su objetivo no es únicamente describir configuraciones o procedimientos, sino preservar el conocimiento técnico necesario para comprender el sistema.

Toda documentación deberá priorizar:

- claridad;
- precisión;
- mantenibilidad;
- reutilización;
- verificabilidad.

---

# Alcance

La documentación deberá representar el estado real del sistema.

No documentar configuraciones hipotéticas ni escenarios no verificados.

Cuando una información no pueda verificarse deberá indicarse explícitamente.

---

# Comprensión

No limitarse a describir qué existe.

Siempre que resulte posible explicar:

- qué es;
- para qué sirve;
- cómo funciona;
- cuándo se utiliza;
- cómo se relaciona con otros componentes;
- qué impacto tiene dentro del sistema.

La documentación deberá facilitar la comprensión del proyecto.

---

# Nivel de detalle

Adecuar el nivel de profundidad al tema documentado.

Evitar tanto:

- documentación excesivamente superficial;
- documentación innecesariamente extensa.

Priorizar la información que resulte útil para comprender y mantener el sistema.

---

# Contexto

Siempre que resulte necesario incorporar contexto suficiente para comprender el tema documentado.

No asumir que el lector conoce:

- el proyecto;
- la arquitectura;
- los procedimientos existentes;
- la documentación previa.

---

# Relaciones

Cuando existan relaciones entre componentes, documentarlas.

Ejemplos:

- dependencias;
- servicios;
- aplicaciones;
- redes;
- bases de datos;
- infraestructura;
- procesos;
- procedimientos.

Favorecer una visión integral del sistema.

---

# Procedimientos

Cuando un documento describa procedimientos deberá explicar:

- objetivo;
- prerrequisitos;
- pasos;
- resultado esperado;
- validaciones;
- posibles errores;
- acciones posteriores.

No limitarse a enumerar comandos.

---

# Arquitectura

Cuando corresponda documentar:

- componentes;
- responsabilidades;
- relaciones;
- dependencias;
- flujos;
- límites del sistema.

Priorizar la comprensión de la arquitectura sobre la enumeración de archivos.

---

# Evolución

La documentación deberá mantenerse sincronizada con el sistema.

Cuando se detecten cambios relevantes:

- actualizar la documentación correspondiente;
- eliminar información obsoleta;
- mantener la coherencia del repositorio.

Evitar documentación desactualizada.

---

# Duplicación

No duplicar conocimiento.

Cuando un tema ya se encuentre documentado:

- reutilizar la documentación existente;
- generar referencias cruzadas;
- ampliar el documento correspondiente cuando resulte apropiado.

---

# Modularidad

Dividir la documentación en documentos pequeños y especializados.

Evitar documentos excesivamente grandes cuando puedan separarse naturalmente.

Cada documento deberá tener una única responsabilidad.

---

# Reutilización

La documentación deberá poder reutilizarse en diferentes contextos.

Siempre que sea posible:

- evitar referencias específicas innecesarias;
- reutilizar conceptos;
- reutilizar procedimientos;
- reutilizar definiciones.

---

# Calidad

Antes de finalizar verificar que la documentación:

- represente correctamente el sistema;
- sea consistente con el resto del repositorio;
- no contenga contradicciones;
- no duplique información;
- mantenga una organización lógica;
- facilite la comprensión del tema documentado.

---

# Principios

Durante toda la generación de documentación deberán respetarse los siguientes principios:

1. Documentar conocimiento antes que archivos.
2. Explicar antes que enumerar.
3. Comprender antes que describir.
4. Representar el estado real del sistema.
5. Mantener una única fuente de verdad.
6. Favorecer la reutilización del conocimiento.
7. Mantener la documentación sincronizada con el proyecto.
8. Facilitar el mantenimiento y la evolución del sistema.
